## 🎯 機能の優先度マトリクス

### 【必須】コア機能（⭐⭐⭐⭐⭐）

```
1. 認証・ログイン
2. シフト希望提出

4. シフト閲覧
5. スタッフ管理
6. セグメント表示（人員配置）
```

**投資判断:** ✅ **これがないと成立しない → SQL化すべき**

```
1. グラフ可視化（タイムライン）

4. 日次・月次シフト表
5. カレンダーDL（iCal）
```

**投資判断:** ✅ **競合との差別化 → SQL化で高速化する価値あり**

```
1. マニュアル管理
```

## 💰 コスト投資の判断基準

### SQL化すべき機能（優先度高）

```
✅ シフト提出・編集・閲覧
   理由: 同時アクセスが発生する
   
✅ スタッフ管理・認証
   理由: データ整合性が重要
   
✅ セグメント表示・グラフ
   理由: 複雑なクエリが必要
   
✅ 除外時間管理
   理由: リアルタイム更新が必要
```

### CSV維持でもOKな機能（優先度低）

```
⚠️ マニュアル管理
   理由: 静的コンテンツ、更新頻度低い
   
⚠️ メモ機能
   理由: 小規模データ、頻度低い
   
⚠️ 在庫管理
   理由: シフトと独立、別システム化も検討
```

## 📊 需要分析サマリー

### 現在のターゲット: **飲食店1店舗**

```
強み:
✅ シフト管理が充実
✅ 可視化が優秀
✅ 除外時間対応

弱み:
⚠️ 機能が多すぎる（在庫管理など）
⚠️ 飲食店特化しすぎ
```

### 公開サービス化するなら

#### パターンA: **汎用シフト管理**

```
残す機能:
✅ シフト提出・編集・閲覧
✅ グラフ・可視化
✅ 除外時間管理
✅ マルチテナント対応

削除する機能:
❌ 在庫管理 → 別サービス化
❌ マニュアル → 別サービス化
❌ メモ機能 → オプション化
```

**ターゲット:** アルバイト雇用がある全業種

#### パターンB: **飲食店特化**

```
全機能維持 + 追加:
✅ 現在の全機能
✅ 予約管理連携
✅ 売上連動シフト最適化
✅ POS連携
```

**ターゲット:** 飲食店のみ

## 💡 結論

### SQL化する価値: ✅ **ある！**

**理由:**
1. **コア機能が充実** - シフト管理として完成度高い
2. **差別化要素あり** - セグメント表示、除外時間など
3. **拡張性** - マルチテナントで複数店舗対応可能
4. **コストは妥当** - ¥3,000/月で始められる

### ただし...

**⚠️ 機能の整理が必要:**
- 在庫管理は分離を検討
- マニュアル管理もオプション化
- シフト管理に集中

**📈 需要の見極め:**
- 汎用化 vs 飲食店特化を決める
- ターゲット市場のサイズを調査
- 競合分析（Airシフト、ジョブカンなど）
